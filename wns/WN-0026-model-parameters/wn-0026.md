# WN 0026 - Model Parameters

- **Status**: *Draft*
- **Discussions-To:** TBD
- **Author:** Michael Toy
- **Experiment-ID:** *modelParameters*

## Motivation

Users want to parameterize Malloy models with values supplied at query time. A primary use case is **row-level access control** — writing a model once where sources carry filters like `where: x.data = $TENANT`, and then supplying the tenant value per API call.

Other use cases: configuration values (max row limits, date cutoffs), session context (user role, locale), and any situation where one compiled model should be reusable across multiple invocations with varying context.

Parameterizing at the model level composes better than parameterizing at the source level. It makes it much easier to write a model where certain filters are applied globally, with each source implementing that filtering as appropriate.

## Declaration

A new top-level statement `parameter:` introduces parameters to a model:

```malloy
parameter:
  TENANT :: string
  MAX_ROWS :: number is 1000
  CUTOFF_DATE :: date is @2024-01-01
```

* `TYPE` is any Malloy atomic type, including compound types (arrays and records).
* The default value of a parameter is an expression which can only include constants and parameter references.

Example declarations showing the range of allowed types:

```malloy
parameter:
  ROLE :: string
  ALLOWED_ROLES :: string[]
  SESSION :: { user_id :: string, tenant :: string }
  AUTH :: ModelDefinedAuthType
```

## Reference

Inside any expression, a parameter is referenced with a leading `$`:

```malloy
source: orders_for_user is orders extend {
  where: orders.tenant_id = $TENANT
}

query: recent_orders is orders_for_user -> {
  where: order_date >= $CUTOFF_DATE
  limit: $MAX_ROWS
}
```

The `$` sigil is a reader cue — it marks a reference as coming from the parameter namespace rather than the model's source/view/field namespace. Parameter names and model names share the declaration namespace, so `source: x is ...` combined with `parameter: x :: string` in the same file is a declaration-time conflict error.

## Scope and Namespaces

Every model has a single flat parameter namespace. All parameters declared in a file are visible to every source, view, query, and expression in that file.

When model A imports model B without selection, B's parameters are brought into A's namespace as well, so any imported source that references `$X` continues to work. The caller of A sees one flat parameter map and supplies values for all parameters, regardless of which file declared them.

**Name collisions between imported parameters are compile-time errors.** If A imports B and C, and both B and C declare a parameter named `MAX_SIZE`, A must disambiguate explicitly via rename-on-import:

```malloy
import { source1, MAX_SIZE as B_MAX } from "b.malloy"
import { source2, MAX_SIZE as C_MAX } from "c.malloy"
```

Two key principles:

1. **Parameters are obligations, not opt-in imports.** If you import a source from B, and that source references a parameter, that parameter must be satisfiable — you can't silently drop it. Parameters always come along with whatever uses them.

2. **Collisions are explicit, never silent.** Two parameters with the same name from different imports force the importer to rename at least one. The language never auto-merges.

## API Shape

Queries expose a `parameters` option on their run methods:

```typescript
query.run({
  parameters: {
    TENANT: "acme",
    MAX_ROWS: 500,
    CUTOFF_DATE: new Date("2024-01-01")
  }
})
```

The parameters argument is a flat JS map keyed by the names the caller sees (i.e., the names in the outer model's namespace, post-rename). Because parameter types are known from the declaration, the API is fully type-checked — TypeScript users get autocomplete and `tsc` errors for wrong parameter shapes.

A parameter is "required" if the query being compiled needs to de-reference it. It will be a runtime error to attempt to run a query with unresolved references.

### Annotations and Introspection

Parameters participate in Malloy's standard annotation system — `#`-annotations distribute over `parameter:` declarations the same way they do over `measure:`, `dimension:`, and other named declaration blocks. No new syntax is needed.

```malloy
parameter:
  # label="Tenant" required
  TENANT :: string

  # label="Order Status" possible_values=[COMPLETE, RETURNED, CANCELLED]
  ORDER_STATUS :: string is "COMPLETE"
```

The foundation API exposes parameter metadata through the same introspection path used for sources and queries — callers can enumerate the parameters a compiled model requires, retrieve their types and defaults, and read their annotations. This is what lets a frontend tool build a parameter-input UI (dropdowns, date pickers, etc.) from the model metadata without the user having to wire anything up manually.

## Relationship to Source Parameters

Prior work on parameters in Malloy: [WN-0002](../WN-0002-parameters/wn-0002.md) (Lloyd's broader original parameters proposal, covering filters, data-function parameters, and physical source parameters) and [WN-0014](../WN-0014-mvp-parameters/WN-0014.md) (the MVP source-parameter implementation, shipped experimentally as `##! experimental.parameters`).

Malloy's current experimental source parameters (`source: foo(x :: string) is ...`) are likely candidates for removal. Model parameters subsume the real-world use cases and are strictly more composable:

- Source parameters require per-instantiation binding at every use site, making multi-source models with shared context tedious.
- Source parameters don't compose across imports.
- The "per-instantiation" binding model is rarely what users actually want in practice — configuration, session context, and RLAC all want "bound once per run, visible everywhere."

Model parameters cover all of these cleanly. If there turns out to be a genuine need for per-instantiation binding (two uses of the same source with different values in one query), that can be revisited later.

## Experimental Status

Like source parameters before them, model parameters ship as experimental under `##! experimental.modelParameters`. The design is untested against real workloads and may need revision — or removal — based on what we learn. The feature will sit behind a compiler flag so early adopters can try it without committing the broader user base.

## Open Questions

These are known open questions to be decided before or during implementation:

- **Evaluation space.** Is `$NAME` a literal (prep-time foldable, usable in schema-shaping positions like `accept:`/`except:`) or an input-space value (usable in filter predicates and expressions, but not in positions that need compile-time constants)? RLAC needs input-space; some cases want literal. Could be one or the other or both, but needs explicit choice.
- **filter&lt;T&gt;** There is a wish for this to also be a legal type for a parameter. Filter expressions can be passed as strings in a parameter block. This is probably all right but I haven't thought it through.

## Comparison with WN-0002

Lloyd's [WN-0002](../WN-0002-parameters/wn-0002.md) proposed a broader parameter feature bundling several capabilities. This proposal addresses some directly, leaves some out of scope, and picks up others as a natural consequence of the model-parameter design.

| Capability | WN-0002 | WN-0026 |
|---|---|---|
| Binding scope | Per-source / per-view / per-dimension / per-measure | Model-wide (one namespace per compilation) |
| Binding timing | At invocation site (`orders(x='foo')`) | At `run()` API call |
| Constant parameters (string, number, date) | Yes | Yes — any Malloy atomic type, including compound types |
| Default values | Yes | Yes |
| Parameterized views | Yes, via per-view parameter slots | Yes, falls out naturally — a view that references `$PARAM` in its body is a parameterized view |
| Parameter annotations for UI hinting | Yes, via `# label=...` on parameter declarations | Yes — uses Malloy's standard annotation system; foundation API exposes parameter metadata for introspection |
| Composes across imports | Not discussed | Yes — namespace rules with rename-on-import |
| RLAC / row-level access control | Possible but awkward (per-source plumbing) | Direct fit (model-wide binding) |
| `filter<T>` as a parameter type | Yes, central to the design | **Open question** — flagged for future work |
| Filter helper functions (`begin_date`, `end_date`, etc.) | Yes, paired with filter types | **Not in scope** — paired with filter-type work |
| Abstract functions (dimensions, measures, views, sources as invocable functions) | Yes | **Not in scope** — orthogonal feature, would need its own WN |
| "Bypass default" semantics (ALL / empty string) | Yes | N/A — callers can supply any value, including ones that neutralize a default's effect |

**Where this proposal is strictly better.** Composition across imports, RLAC ergonomics, and the overall "bound once, visible everywhere" model for shared context — all direct consequences of scoping at the model level rather than per-declaration.

**Where this proposal is intentionally smaller.** `filter<T>` parameters and abstract functions are meaningful features on their own. Bundling them into this WN would expand the scope significantly without improving the core model-parameter design. Both are better addressed as follow-on WNs once model parameters have shipped and we've seen how they're used in practice.
