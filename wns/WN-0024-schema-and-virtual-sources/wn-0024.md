# WN 0024 - Struct Declarations and Virtual Sources

- **Status**: *Experimental*
- **Discussions-To:** https://github.com/malloydata/malloy/pull/2729
- **Author:** Michael Toy
- **Experiment-ID:** virtual_source

## Overview

This WN describes two features that share a common syntax element — the `struct:` declaration — but address different problems.

**Struct declarations on table sources** make a Malloy source self-describing. Today, understanding a source requires access to the underlying database — the schema lives in the database, not in the `.malloy` file. By declaring a struct and applying it to a source, the source becomes a complete, readable document. An LLM can reason about the source and write queries against it without a database connection. A CI pipeline can compile and validate a model without database credentials. A human reading the file can see exactly what fields exist without running a query.

**Virtual sources** define a source with no underlying table. The struct declaration is the *only* source of fields — there is no database to introspect. The actual table is resolved at query time through a mapping, making it possible to write models against tables that don't exist yet, or to point the same model at different tables in different environments.

Both features use named, reusable struct objects applied to sources via `extend<struct_name>`. On a table source, the struct validates and filters. On a virtual source, the struct defines.

WN-0013 explored earlier thinking on this topic. This WN supersedes it with a concrete design and implementation.

## Structs

### Declaring a Struct

A `struct:` declaration is a top-level statement, like `source:` or `query:`. It defines a named collection of typed fields:

```malloy
##! experimental.virtual_source

struct: store is {
  id :: number,
  name :: string,
  street :: string,
  city :: string,
  state :: string
}
```

A struct is a bag of typed fields — it has no connection to any database and no behavior on its own. It becomes meaningful when applied to a source or referenced from another struct.

### Nesting Structs

Struct fields can reference other named structs, allowing complex nested structures to be composed from reusable pieces:

```malloy
struct: address is {
  street :: string,
  city :: string,
  state :: string,
  zip :: string
}

struct: person is {
  name :: string,
  home :: address
}

struct: store is {
  id :: number,
  name :: string,
  location :: address,
  manager :: person
}
```

A field declared as a named struct behaves the same as an inline record type — it describes a nested record in the data. The difference is purely ergonomic: named structs can be reused across multiple declarations.

### Extending a Struct

A struct can incorporate the fields of another struct using `extend`:

```malloy
struct: cigar_store is store extend {
  humidor_capacity :: number,
  walk_in :: boolean
}
```

This produces a struct with all the fields of `store` plus the new fields. There is no hierarchy or inheritance relationship — `cigar_store` is not a subtype of `store`. It's simply a struct that happens to include the same fields, plus more. Think of it as duck typing: the resulting struct is a flat bag of fields regardless of how it was assembled.

### Applying Structs to Sources

The `extend<>` syntax on a source accepts a comma-separated list of structs and declares the **complete expected schema** of that source:

```malloy
struct: address_fields is {
  street :: string,
  city :: string,
  state :: string,
  zip :: string
}

struct: store_fields is {
  id :: number,
  name :: string,
  store_type :: string
}

source: stores is duckdb.table('stores.parquet') extend<store_fields, address_fields> {
  dimension: full_address is concat(street, ', ', city, ', ', state)
}
```

There is an important semantic distinction between the three uses of `extend` in this design:

- **`struct: b is a extend { ... }`** — adds fields to a struct, building up a larger struct
- **`source: b is a extend { ... }`** — adds dimensions, measures, joins to a source, building up
- **`source: b is a extend<s> { ... }`** — declares the complete expected schema of the source

The first two are additive. The third is an assertion about the full shape. The struct in `extend<>` describes what the source's public intrinsic fields should be — fields not listed become private, type mismatches are errors. This is true regardless of where the source's fields come from (a database table, a virtual mapping, or a parent source):

```malloy
struct: full_schema is { id :: number, name :: string, city :: string }
struct: narrow_schema is { id :: number, name :: string }

source: a is duckdb.virtual('x') extend<full_schema> { }

// b has all of a's fields, but only id and name are public
source: b is a extend<narrow_schema> { }
```

Because `extend<>` always means "the complete expected schema," composing multiple structs at the struct level or at the source level are equivalent:

```malloy
// Style 1: compose at the source
struct: base_fields is { id :: number, name :: string }
struct: metric_fields is { revenue :: number, cost :: number }

source: report_a is duckdb.virtual('report') extend<base_fields, metric_fields> { }
```

```malloy
// Style 2: extend at the struct
struct: report_fields is base_fields extend {
  revenue :: number,
  cost :: number
}

source: report_b is duckdb.virtual('report') extend<report_fields> { }
```

Both produce the same result.

### Supported Types

Struct declarations support the full range of Malloy types:

| Type | Example |
|------|---------|
| Basic types | `x :: string`, `y :: number`, `z :: boolean`, `d :: date`, `t :: timestamp` |
| Timestamp with timezone _(if supported by the databgase)_ | `t :: timestamptz` |
| Arrays | `tags :: string[]`, `scores :: number[]` |
| Inline records | `address :: { street :: string, city :: string }` |
| Named structs | `location :: address`, `manager :: person` |
| Arrays of records | `items :: { name :: string, qty :: number }[]` |
| SQL native types | `big_id :: "BIGINT"`, `geo :: "GEOGRAPHY"` |

SQL native types are written as quoted strings. This handles database-specific types that don't have a Malloy equivalent. A struct field declared as `"BIGINT"` will match a column whose database type maps to `"BIGINT"` in Malloy's type system.

## Structs on Table Sources

The previous section described the semantics of `extend<>`: it declares the complete expected schema. On a table source, the compiler enforces this against the actual table. Specifically:

**1. Hides undeclared fields.** Any intrinsic field (a column from the underlying table) not listed in the struct is marked `private`. It still exists — joins and expressions can reference it — but it won't appear in query output or auto-complete unless explicitly re-exposed.

```malloy
struct: airport_fields is {
  code :: string,
  elevation :: number
}

source: airports is duckdb.table('airports.parquet') extend<airport_fields> {
  // full_name, city, state, etc. are now private
  // code and elevation remain public
}
```

**2. Validates types.** If a declared field exists in the source but its type doesn't match, the compiler reports an error on the specific field declaration:

```malloy
struct: bad_fields is {
  elevation :: string  // actual column is number — error!
}

source: airports is duckdb.table('airports.parquet') extend<bad_fields> { }
// Error: Schema type mismatch for 'elevation': expected string, got number
```

**3. Validates existence.** If a declared field doesn't exist in the source at all, the compiler reports an error:

```
Field 'nonexistent' declared in struct does not exist in source
```

### What Structs Do Not Affect

- **Dimensions, measures, and joins** defined in the `extend` block are unaffected by the struct. The struct only governs intrinsic (table-derived) fields.
- Structs do not create fields on table sources — they filter and validate the fields that already exist from the underlying table.

### Use Cases

**Narrowing a wide table:**

```malloy
struct: event_fields is {
  event_id :: string,
  event_type :: string,
  event_ts :: timestamptz,
  user_id :: string,
  payload :: json
}

source: events is bq.table('analytics.raw_events') extend<event_fields> {
  // The raw table has 80+ columns from various SDKs.
  // Only these five are part of the modeled interface.
}
```

**Catching upstream drift:**

```malloy
// If the data team changes `user_id` from string to number,
// or renames `event_ts` to `event_time`, the model fails
// at compile time instead of producing wrong results.
```

**Self-describing sources for AI workflows:**

```malloy
// An LLM can read this file and write queries against it
// without needing a database connection or credentials.
struct: order_fields is {
  order_id :: string,
  customer_id :: string,
  order_date :: date,
  total :: number,
  status :: string
}

source: orders is duckdb.table('orders.parquet') extend<order_fields> {
  measure: total_revenue is total.sum()
  measure: order_count is count()
}
```

## Virtual Sources

### Motivation

Struct declarations validate an existing table's shape. Virtual sources go one step further: they define a source with *no* underlying table. The struct provides the entire field list, and the actual table is resolved at query time through a **virtual map**.

This separation is useful when:

- The table will be created by a pipeline that hasn't run yet, but you want to write and compile the model now
- The same model should run against different tables in different environments
- You're building a generic model that will be parameterized by table at runtime

### Syntax

```malloy
##! experimental.virtual_source

struct: user_facts_fields is {
  user_id :: string,
  signup_date :: date,
  lifetime_value :: number,
  segment :: string
}

source: user_facts is duckdb.virtual('user_facts') extend<user_facts_fields> { }
```

The `virtual('name')` call replaces `table('path')` or `sql("""...""")` as the source's origin. The string argument is a logical name used to look up the actual table at query time.

### Structs on Virtual Sources

On a table source, `extend<>` validates against existing fields. On a virtual source, there are no existing fields — the struct *defines* them. The declared fields become intrinsic fields on the source, behaving like table columns rather than computed dimensions. There is no type checking or existence checking — the declared fields are the source's entire schema.

A virtual source without a struct is legal but not useful — it has no fields.

```malloy
// Full virtual source with struct and extensions
struct: order_fields is {
  order_id :: string,
  order_date :: date,
  total :: number
}

source: orders is duckdb.virtual('orders') extend<order_fields> {
  // You can extend virtual sources like any other
  dimension: order_year is year(order_date)
  measure: total_revenue is total.sum()
}
```

### The Virtual Map

At query time, the application provides a **virtual map** that resolves virtual names to actual table paths:

```typescript
type VirtualMap = Record<string, Record<string, string>>;
// { [connectionName]: { [virtualName]: tablePath } }
```

For example:

```typescript
const virtualMap = {
  duckdb: {
    user_facts: 'analytics.user_facts_v2',
    orders: 'sales.orders_2024'
  }
};
```

The virtual map is provided the same way as a build manifest — through `PrepareResultOptions` or loaded from configuration. This is a deliberate parallel: just as a manifest maps persistent sources to pre-built tables, a virtual map maps virtual sources to concrete tables.

### Missing Map Entry

If a virtual source is used in a query and the virtual map has no entry for it, the compiler produces a hard error. Unlike persistence (where a missing manifest entry means "expand inline"), there is no fallback — a virtual source with no table mapping cannot generate SQL.

```
No virtual map entry for 'user_facts' on connection 'duckdb'
```

This is intentional. A virtual source has no SQL of its own. If the map doesn't tell us where to find the data, there's nothing we can do.

### Use Cases

**Environment-specific tables:**

```malloy
struct: feature_fields is { user_id :: string, feature_vec :: string }

// The model is the same everywhere
source: features is bq.virtual('features') extend<feature_fields> { }

// Dev config maps to a sample table
// { "bq": { "features": "dev.features_sample" } }

// Prod config maps to the full table
// { "bq": { "features": "prod.ml_features_v3" } }

// Test config maps to a fixture
// { "bq": { "features": "test_fixtures.features_small" } }
```

**Pipeline output:**

```malloy
struct: daily_agg_fields is {
  date :: date,
  total_users :: number,
  total_revenue :: number
}

// Model written before the pipeline exists
source: daily_agg is duckdb.virtual('daily_agg') extend<daily_agg_fields> {
  measure: avg_revenue_per_user is total_revenue.sum() / total_users.sum()
}

// When the pipeline runs, it writes to a table and updates the virtual map
```

## Summary

| Source Type | Struct Behavior | Fields Come From |
|-------------|----------------|------------------|
| `table()` without struct | All table columns are public | Database |
| `table()` with struct | Declared columns public, rest private, types validated | Database (validated) |
| `virtual()` with struct | Declared columns are the only fields | Struct declaration |
| `virtual()` without struct | Empty source (legal but useless) | Nothing |

## Future Direction: Structs as First-Class Types

The initial implementation uses structs for source schema control and virtual source field definitions. However, structs are a natural foundation for named types throughout the Malloy language.

As a concrete example, a named struct could be used as a cast target in expressions:

```malloy
struct: address is {
  street :: string,
  city :: string,
  state :: string
}

source: orders is duckdb.table('orders.parquet') extend {
  dimension: shipping_addr is raw_shipping::address
}
```

Here `::address` behaves the same as casting to the equivalent inline record type `::{ street :: string, city :: string, state :: string }`, but is more readable and stays in sync with the struct definition.

More broadly, anywhere Malloy currently accepts a type — field declarations, casts, function signatures — a named struct could eventually appear. This is not part of the initial implementation, but the design is intended to support this evolution.

## Relationship to Other Features

**Persistence (WN-0022):** Virtual sources and persistent sources are complementary. A persistent source starts with real data and caches it. A virtual source starts with no data and maps it in at runtime. Both use the same pattern of compile-time IR + runtime resolution through a map.

**Connection Config (WN-0023):** The virtual map will be loadable from `malloy-config.json`, following the same pattern as the build manifest.

## Current Limitations

- Structs cannot change a field's type on a table source — they can only validate that it matches
- Struct extending struct always adds fields — narrowing or renaming fields is the domain of `include`
- Virtual sources do not support schema introspection (there is no table to introspect)
- The virtual map is flat (name to table path) — it does not support parameterized mappings
