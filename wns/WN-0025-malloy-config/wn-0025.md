# WN 0025 - Malloy Configuration

- **Status**: *Accepted*
- **Discussions-To:** TBD
- **Author:** Michael Toy

## Overview

Malloy tools need to agree on database connection configurations and other runtime settings. The Malloy config system provides a unified way to declare connections, resolve secrets and context-dependent values, and discover configuration files across environments.

The system is designed around four user types, each adding complexity only when needed:

- **Soloist** -- Opens VS Code, creates Malloy files, connects to data. Never has to think about config.
- **Shop** -- Small team, shared codebase in git, one or two databases. Credentials stay out of git.
- **Developer** -- Working on Malloy code for Publisher or similar. Needs to test locally with one set of connections, publish with different ones.
- **Publisher** -- Hosts packages as a web service. Source of truth is a database, not config files.

The core implementation lives in `@malloydata/malloy` and shipped in version 0.0.373.

## The Configuration File

Configuration is authored as JSON files named `malloy-config.json`. A minimal configuration declares one connection:

```json
{
  "connections": {
    "duckdb": {
      "is": "duckdb"
    }
  }
}
```

A more complete example:

```json
{
  "connections": {
    "warehouse": {
      "is": "bigquery",
      "projectId": {"env": "GCP_PROJECT"},
      "location": "US"
    },
    "logs": {
      "is": "postgres",
      "host": {"env": "PG_HOST"},
      "port": 5432,
      "databaseName": "analytics",
      "username": {"env": "PG_USER"},
      "password": {"env": "PG_PASSWORD"}
    },
    "local": {
      "is": "duckdb"
    }
  }
}
```

The `connections` object declares named database connections. Malloy models reference connections by name (e.g., `source: foo is warehouse.table('orders')`). The first connection listed is the default.

Each connection has an `is` property identifying the backend, plus backend-specific properties. Supported backends and their properties are documented by the connection packages themselves.

### Other Top-Level Fields

| Field | Description |
|-------|-------------|
| `connections` | Named connection entries |
| `manifestPath` | Where persistence manifests are stored (default: `"MANIFESTS"`) |
| `virtualMap` | Map of virtual source names to table references |
| `includeDefaultConnections` | When `true`, fabricate a connection for every registered backend type not already declared |

## Overlay References

Any connection property value can be replaced with an **overlay reference** -- a single-key object where the key names the source and the value is the path into it:

- `{"env": "PG_PASSWORD"}` -- environment variable
- `{"config": "rootDirectory"}` -- host context
- `{"session": "user_id"}` -- session data (host-registered)
- `{"secret": "db_password"}` -- secret store (host-registered)

This generalizes the original `{"env": "VAR"}` pattern. The single-key constraint distinguishes references from literal objects.

### Built-in Overlays

Two overlays are always available:

- **`env`** -- Environment variables. `{"env": "PG_PASSWORD"}` resolves to `process.env.PG_PASSWORD`.
- **`config`** -- Host context. By default a no-op (`() => undefined`). Discovery-driven hosts populate it with `rootDirectory` (the project root) and `configURL` (where the config file was found).

### Unresolved References

References can fail to resolve in three ways, none of which is a hard error:

1. **Unknown overlay source** (e.g., `{"zzz": "foo"}` with no `zzz` overlay) -- warning logged, property dropped.
2. **Known overlay returns `undefined`** (e.g., env var not set) -- silent drop. If the property was required, the connection factory complains at lookup time.
3. **Unresolved reference inside a property default** -- silent drop. A default is a hint, not a requirement.

A broken connection entry should not prevent querying through other connections.

## Connection Registry

The registry is a three-layer system:

**1. Core (`@malloydata/malloy`)** -- Owns the registry. Stores registered connection types, compiles config, creates connections on demand.

**2. Connection packages** -- Each `@malloydata/db-*` package registers itself as a side effect of being imported. Importing `@malloydata/db-duckdb` registers the `duckdb` type with its factory and property metadata.

**3. `@malloydata/malloy-connections`** -- Convenience package that imports all database packages for side-effect registration.

### Property Defaults

Every connection property in the registry can declare a `default` value -- either a literal or an overlay reference. If a connection entry doesn't set that property, the registered default fills in. This is uniform and applies to every connection the resolver sees.

For example, DuckDB's `workingDirectory` defaults to `{"config": "rootDirectory"}`, so DuckDB automatically resolves relative file paths against the project root without the user ever mentioning `workingDirectory`.

### `includeDefaultConnections`

Independently from property defaults, setting `includeDefaultConnections: true` tells the resolver to fabricate one connection per registered backend type not already declared. The fabricated entries then get property defaults applied like any other entry.

A soloist who writes `{"includeDefaultConnections": true}` and nothing else gets working connections for every registered backend.

These two mechanisms are orthogonal: property defaults control what properties each entry has; `includeDefaultConnections` controls whether the entry exists in the first place.

## The MalloyConfig Class

`MalloyConfig` is the resolved config object that Runtime consumes. Construction is synchronous with zero IO.

```typescript
// Parse a config file
new MalloyConfig(jsonString)

// In-memory config + overlays
new MalloyConfig(configObject, overlays?)
```

### Resolution Pipeline

```
JSON/POJO -> CompiledConfig -> overlays -> MalloyConfig -> Runtime
```

1. The constructor compiles the POJO into a typed dictionary tree (schema-directed walk, references become explicit nodes).
2. The resolver walks the tree, calls the matching overlay for each reference, produces a plain resolved object.
3. Property defaults are applied to every connection entry. If `includeDefaultConnections` is set, entries are fabricated for missing backend types first.
4. Connections are created via the connection registry from the resolved entries.
5. The result is a `MalloyConfig` ready for Runtime.

### Section Compilers

The compiler is not a generic recursive walker. Each top-level key has a **section compiler** that knows the shape of that section and controls where overlay references are allowed. `virtualMap` entries are literal data; `{"env": "X"}` inside `virtualMap` is not a reference. This keeps the security invariant sharp -- no generic reference expansion, no injection into literal slots.

### Connection Lookup Wrapping

`MalloyConfig` produces a `LookupConnection` that apps can pass directly to Runtime. Apps that need customization use `wrapConnections()`:

```typescript
// VS Code -- settings as fallback
config.wrapConnections(base => name =>
  base(name) || settingsLookup(name) || defaultLookup(name)
);

// Publisher -- attach DuckDB federation
config.wrapConnections(base => name => {
  const c = base(name);
  if (hasAttachments(name)) attachDatabases(c);
  return c;
});
```

### Releasing Connections

`runtime.releaseConnections()` walks the connection cache and calls `close()` on each. Connections never looked up were never constructed and have nothing to release.

## Config Discovery

For local hosts (VS Code, CLI, AI tools), a discovery helper finds config files:

```typescript
const config = await discoverConfig(startURL, ceilingURL, urlReader);
```

Discovery walks upward from `startURL` to `ceilingURL` looking for `malloy-config.json` and an optional `malloy-config-local.json`. On a hit, it returns a fully-constructed `MalloyConfig` with the `config` overlay populated (`rootDirectory` = the ceiling, `configURL` = the matched file). On a miss, it returns `null`.

### Local Config Override

Developers can create `malloy-config-local.json` (gitignored) alongside the shared config. If both files are found at the same directory level, the local file supersedes the shared file -- local is the complete config with no merging. The local file is for credentials, local paths, and other developer-specific settings.

### Broken Config Files

A file that isn't present at a directory level is normal -- the walker moves on. A file that is present but cannot be parsed as JSON is a hard error. Silently skipping a broken file would hide user mistakes.

## Build Manifests

When a model uses persistent sources (WN-0022), the Runtime needs a build manifest for table substitution. Three ways to provide one:

- **Auto-read (default)** -- If the config has a `configURL`, `MalloyConfig` computes `manifestURL` as `{manifestPath}/malloy-manifest.json` relative to the config file. Runtime lazily reads it on the first persistence query.
- **Explicit constructor option** -- Pass `buildManifest:` to Runtime's constructor.
- **Setter after construction** -- `runtime.buildManifest = manifest`.

## Host Adoption

### VS Code

Three-level config resolution with strict priority:

1. **Project config** -- discovered by walking up from the active file to the workspace root.
2. **Global config** -- if no project config and `globalConfigDirectory` is set.
3. **Settings + defaults** -- legacy/soloist fallback using `includeDefaultConnections: true` with settings connections layered via `wrapConnections`.

The presence of a config file is the trust boundary. When a config file is active, settings connections do not leak in unless the file opts in via `includeDefaultConnections: true`.

### CLI

Adds `--projectDir <dir>` for VS Code-compatible config discovery. Three modes:

| Mode | Behavior |
|------|----------|
| neither flag | Global config at `~/.config/malloy/malloy-config.json` |
| `--config <path>` | Use that specific config file, no walk-up |
| `--projectDir <dir>` | Walk up from cwd to `<dir>` looking for config |

### Publisher

Publisher's source of truth is its database. It assembles a POJO from DB state, constructs `MalloyConfig` with custom overlays (`config`, `session`), and uses `wrapConnections` for DuckDB ATTACH federation. Publisher ignores per-package `malloy-config.json` files.

## Usage Examples

```typescript
// Soloist -- no config file, just defaults
const config = new MalloyConfig({includeDefaultConnections: true});
const runtime = new Runtime({config, urlReader});

// Explicit config
const config = new MalloyConfig({
  connections: {
    warehouse: {is: 'duckdb', databasePath: '/data/wh.duckdb'},
  },
});
const runtime = new Runtime({config, urlReader});

// Discovery (local hosts)
const config =
  (await discoverConfig(startURL, projectRootURL, urlReader)) ??
  new MalloyConfig(
    {includeDefaultConnections: true},
    {config: contextOverlay({rootDirectory: projectRootURL.toString()})}
  );
const runtime = new Runtime({config, urlReader});

// Publisher -- per-request overlays
const config = new MalloyConfig(publisher.buildConfig(pkgId), {
  config: publisherOverlay(pkgId),
  session: sessionOverlay(request),
});
const runtime = new Runtime({config, urlReader});
```

## Relationship to WN-0023

WN-0023 describes how **connections** work -- the connection registry, how backends register themselves via side-effect imports, how connection entries declare properties with typed metadata, and how factories create live connections from config entries.

This WN describes how **configuration** works -- how config is authored as JSON, discovered by walking directory trees, compiled into a typed dictionary, resolved through overlays, and delivered to Runtime as a `MalloyConfig` instance. The connection registry (WN-0023) is one component of the configuration pipeline described here.
